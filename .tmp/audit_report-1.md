Highest-Risk Findings

Blocker: the repository is not statically verifiable as delivered. The backend manifest defines only a binary target, while the documented test path uses cargo test --lib, and the integration tests import fleetreserve_backend::... as if a library crate exists. See repo/backend/Cargo.toml (line 1), repo/run_tests.sh (line 7), repo/backend/tests/integration_tests.rs (line 11), and the absence of lib.rs in repo/backend/src. The frontend also uses serde_wasm_bindgen without declaring it in repo/frontend/Cargo.toml (line 9), while pages call it in repo/frontend/src/pages/login.rs (line 21) and repo/frontend/src/pages/reservations.rs (line 21). The backend backup handler imports rusqlite::backup::Backup in repo/backend/src/handlers/backup.rs (line 24) but rusqlite is declared only with bundled in repo/backend/Cargo.toml (line 15).
Blocker: backup and restore are not implemented to the prompt. create_backup copies into an in-memory DB, never writes an encrypted SQLite backup, then stores metadata with size 0 and empty hash; restore_backup only checks path existence and logs “Restore initiated”. See repo/backend/src/handlers/backup.rs (line 24).
High: CSRF protection is both weak and inconsistently applied. require_csrf accepts any non-empty header and explicitly says tokens are not stored server-side, so requests are not session-bound; POST /api/uploads omits CSRF checking entirely. See repo/backend/src/handlers/auth.rs (line 134) and repo/backend/src/handlers/uploads.rs (line 14).
High: object-level/store authorization is missing on multiple state-changing flows. Reservation creation trusts client-supplied store_id and asset_id with no store ownership check in either handler or engine; bay listing/creation accept arbitrary store IDs; ticket redeem/undo never verify that the staff member belongs to the ticket’s store; uploads accept caller-supplied store_id. See repo/backend/src/handlers/reservations.rs (line 9), repo/backend/src/services/reservation_engine.rs (line 84), repo/backend/src/handlers/bays.rs (line 18), repo/backend/src/handlers/tickets.rs (line 36), and repo/backend/src/handlers/uploads.rs (line 29).
High: several prompt-critical flows are stubbed or mismatched. The ticket view renders a QR placeholder instead of a QR code in repo/frontend/src/components/ticket_display.rs (line 15). The check-in UI claims “ticket number or ID” but posts the raw input to /tickets/{id}/redeem, while the backend only resolves tickets by ID, not by ticket number, in repo/frontend/src/pages/tickets.rs (line 51) and repo/backend/src/services/ticket_engine.rs (line 53). The reservation form submits datetime-local values without seconds, but the backend requires %Y-%m-%dT%H:%M:%S, so the documented reservation path is statically inconsistent: repo/frontend/src/pages/reservations.rs (line 86) vs. repo/backend/src/services/reservation_engine.rs (line 19).
High: calendar day/week/status requirements are not actually implemented. The frontend sends view, but the backend always builds a single-day 15-minute grid and ignores asset_status; the UI status filter is purely local state and never reaches the API. See repo/frontend/src/pages/calendar.rs (line 15), repo/backend/src/models/mod.rs (line 441), and repo/backend/src/handlers/calendar.rs (line 35).
High: the audit trail is tamper-evident, but not append-only as required. The schema defines a normal table with no immutability guard, and the tests themselves update rows in place to simulate tampering. See repo/backend/migrations/001_initial_schema.sql (line 149) and repo/backend/src/audit/chain.rs (line 135).
Medium: session handling is absolute-expiry only, not a 12-hour idle timeout with refresh. Token creation sets exp once, validation only compares exp to now, and no request path reissues tokens. See repo/backend/src/auth/session.rs (line 17) and repo/frontend/src/api/client.rs (line 57).
Medium: ticket redemption does not enforce the reservation validity window; it only checks whether the ticket was previously redeemed/undone. See repo/backend/src/services/ticket_engine.rs (line 59).
Medium: the test and reviewer docs overstate coverage. API_tests/*.rs and unit_tests/*.rs are mostly narrative comments, docs/test-coverage.md claims tests that do not exist, and docs/reviewer-notes.md references an AuthUser extractor and route auth middleware that are not present. See docs/test-coverage.md (line 11), repo/API_tests/test_api_auth.rs, and docs/reviewer-notes.md (line 47).
Security Summary

Authentication entry points: Partial Pass. Password hashing and signed tokens exist, but session timeout semantics do not match the prompt, and startup seeds a default admin account in repo/backend/migrations/002_seed_data.sql (line 10).
Route-level authorization: Fail. Route grouping in repo/backend/src/routes/mod.rs (line 19) does not attach auth/role middleware despite docs claiming it.
Object-level authorization: Fail. Cross-store/state-changing gaps exist in reservations, bays, tickets, and uploads.
Function-level authorization: Partial Pass. Many handlers call require_role, but unsupported-role cases can still degrade into incorrect behavior, such as list_assignments returning internal-error flow for disallowed roles in repo/backend/src/handlers/assignments.rs (line 16).
Tenant/store isolation: Fail. Read paths are mixed; several write paths trust caller-controlled store_id.
Admin/internal/debug protection: Partial Pass. Admin handlers do call require_role, but restore is a stub and backup is nonfunctional.
Tests And Logging

Unit tests: Partial Pass. Useful inline unit tests exist in backend modules, but they do not cover several critical authorization and prompt-fit gaps.
API / integration tests: Fail. The advertised API suites are mostly commented specifications, and backend integration tests depend on a library crate shape the manifest does not define.
Logging / observability: Partial Pass. tracing and audit logging are present, but claims like route middleware and idle-session handling are not backed by equivalent instrumentation.
Sensitive-data leakage risk: Partial Pass. Vehicle list/get paths mask VIN/plate, but audit logs and backup metadata still record operator-chosen paths and action details; manual review is still required for runtime logs.
